# 通用记忆系统设计

## 现状分析

从 mes-agent、english-agent、memory-agent 三个项目的实际使用情况来看，当前记忆系统的问题：

### 当前做法
```
memory/
├── 2026-06-06.md    # 自由格式的日记式日志
├── 2026-06-09.md    # 同上
└── ...
MEMORY.md            # 手动维护的长期记忆
```

### 暴露的问题

| 问题 | 表现 | 根因 |
|------|------|------|
| **事实难提取** | 想查"用户偏好"得全文搜索 | 日记式叙事，事实混在故事里 |
| **无实体模型** | 无法问"关于项目X我知道什么" | 没有按实体组织信息 |
| **跨会话断裂** | 新会话得从头 memory_search | 没有会话摘要和关键事实提取 |
| **信息平权** | 架构决策和"今天天气不错"同级 | 没有优先级/类型区分 |
| **无生命周期** | 旧日志无限堆积 | 没有合并/归档/清理机制 |
| **手动维护** | MEMORY.md 靠 AI 自觉更新 | 没有触发条件和更新契约 |

---

## 三层记忆架构

```
┌─────────────────────────────────────────────────────┐
│                  Layer 1: MEMORY.md                  │
│  永久知识库 — 结构化、可查询、主动维护               │
│  ├─ 用户偏好（User Preferences）                     │
│  ├─ 项目状态（Project State）                        │
│  ├─ 决策记录（Decision Records）                     │
│  ├─ 已知问题（Known Issues）                         │
│  └─ 关键事实（Key Facts）                            │
├─────────────────────────────────────────────────────┤
│                  Layer 2: memory/*.md                │
│  会话日志 — 结构化记录，末尾含「关键事实提取」        │
│  ├─ 会话概要                                         │
│  ├─ 做了什么                                         │
│  ├─ 关键事实（→ 自动喂入 Layer 1）                   │
│  └─ 待办/待跟进                                      │
├─────────────────────────────────────────────────────┤
│               Layer 3: memory/entities/              │
│  实体注册表 — 每个实体一个文件，快速定位              │
│  ├─ users/      用户信息、偏好、open_id 映射          │
│  ├─ projects/   项目状态、配置、关键决策              │
│  ├─ tools/      工具配置、API 凭证、连接信息          │
│  └─ concepts/   领域概念、术语、规则                  │
└─────────────────────────────────────────────────────┘
```

|---------|------|

## Layer 2: memory/*.md — 会话日志

### 文件结构

```markdown
# 2026-06-11 日志

## 📋 会话概要
- 时间：14:00-16:30
- 主题：记忆系统重构设计
- 参与人：用户 + generator-agent

## ✅ 完成事项
1. 分析 mes-agent/english-agent 现有记忆系统的问题
2. 设计三层记忆架构
3. 编写通用记忆系统模板

## 🔑 关键事实（→ 需写入 MEMORY.md）
- 当前记忆系统的问题是：日记式叙事、无实体模型、跨会话断裂
- 三层架构：MEMORY.md（永久）+ memory/*.md（会话）+ memory/entities/（实体）
- 实体注册表按 users/projects/tools/concepts 分类

## 📝 决策记录
- 无（本次为设计讨论，未做实施决策）

## ⏳ 待办/待跟进
- [ ] 将本设计写入框架生成器的 templates/skills/memory-system-skill/
- [ ] 在 mes-agent 试点实施

## 💡 备注
- 用户提到"不在飞书群里互动一次找不到对应群"这个痛点
- 通知系统需要先 webchat 后飞书群
```

### 关键规则

1. **每个会话写一个文件**，以日期命名 `YYYY-MM-DD.md`
2. **末尾必须包含「关键事实」** 区块，这是喂入 Layer 1 的接口
3. **会话开始时先读 MEMORY.md**，了解上下文后再开始工作
4. **会话结束时更新 MEMORY.md**，将关键事实合并进去

### 维护规则

| 触发条件 | 动作 |
|---------|------|
| 首次提到新实体 | 创建对应实体文件 |
| 实体状态变更 | 更新实体文件 |
| 用户提供个人信息 | 更新用户实体 |
| 工具配置变更 | 更新工具实体 |
| 实体长期未更新 | 标记为"可能过时" |

## 三层的协作流程

```
会话开始
  │
  ├→ 读取 MEMORY.md  → 了解永久知识（偏好、项目状态、决策）
  ├→ 读取 entities/  → 了解当前活跃实体的详细状态
  └→ 开始工作
       │
       ▼
  工作中产生新信息
       │
       ├→ 用户偏好 → 立即写入 MEMORY.md
       ├→ 决策 → 写入 MEMORY.md 决策记录 + entities/ 对应实体
       ├→ 项目状态变更 → 更新 entities/ 对应实体
       └→ 零散事实 → 记录在会话日志的关键事实区块
       │
       ▼
  会话结束
       │
       ├→ 写会话日志（memory/YYYY-MM-DD.md）
       ├→ 从关键事实区块提取 → 更新 MEMORY.md
       └→ 更新 entities/ 中变更的实体
```

## 解决的问题对照

| 问题 | 解决方案 | 对应机制 |
|------|---------|---------|
| 事实难提取 | 按分类存放，不是日记式叙事 | MEMORY.md 结构化分类 |
| 无实体模型 | 每个实体独立文件，快速定位 | memory/entities/ |
| 跨会话断裂 | 会话开始先读 MEMORY.md | 启动流程约束 |
| 信息平权 | 决策记录有独立格式（含日期+理由） | ADR 格式 |
| 无生命周期 | 关键事实区块 → 合并到 MEMORY.md | 会话结束更新契约 |
| 手动维护 | 触发条件驱动，不是靠自觉 | 维护规则表 |

## 与现有框架生成器的集成

此设计可作为框架生成器的第 3 个内置 Skill 模板：

```
templates/skills/memory-system-skill/SKILL.md
```

创建项目时通过 `引入技能:memory-system` 自动生成：

```
memory/
├── MEMORY.md              # 永久知识库（预填分类模板）
├── 2026-06-11.md          # 会话日志（首次会话自动创建）
└── entities/
    ├── users/
    ├── projects/
    ├── tools/
    └── concepts/
```

## 迁移路径

### 对现有项目

```
1. 创建 memory/entities/ 目录
2. 从现有 MEMORY.md 和 memory/*.md 中提取实体信息
3. 为每个活跃实体创建文件
4. 重构 MEMORY.md 为结构化格式
5. 后续会话按新规范执行
```

### 对新项目

```
1. 框架生成器创建时自动生成三层结构
2. 首次会话自动写入初始状态
3. 按维护规则持续更新
```
