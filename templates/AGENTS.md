## 交互协议

### 用户 → Agent

所有用户指令通过 OpenClaw 交互通道发送，由主Agent统一接收和路由。
详见 `TOOLS.md`。

### Agent → 脚本

Agent 通过 `exec` 命令调用业务脚本，遵循 `ARCHITECTURE.md` 定义的集成契约。
详见 `ARCHITECTURE.md` 第4节。

### Agent → 存储

Agent 通过数据操作层访问存储，不直接操作底层存储。
存储配置见 `datas/config.json` 的 `storage` 字段。

---

## 架构约束

1. **职责单一**：每个 Agent/脚本只做一件事
2. **数据流单向**：数据按 `ARCHITECTURE.md` 定义的方向流动，不反向依赖
3. **错误向上传递**：脚本错误 → 调用方 Agent 处理 → 主Agent 兜底 → 用户
4. **配置集中**：所有参数统一在 `datas/config.json` 管理
5. **状态机驱动**：实体状态严格按 `ARCHITECTURE.md` 定义流转

---

## 脚本架构

脚本分类、调用编排、输出格式规范详见 `SCRIPTS.md`。

### 脚本调用链路

```
用户指令
  └→ Agent 解析
       ├→ 直接处理（简单查询/状态变更）
       └→ 调用脚本（exec）
            ├→ cli 模式：每次调用启动子进程
            ├→ daemon 模式：通过 IPC 与常驻进程通信
            └→ one-shot 模式：独立运行后退出
                 └→ 返回结果 → Agent 组装 → 用户
```

### 编排规则

- **无依赖则并行**：多个独立查询可同时发起
- **有依赖则串行**：后一步依赖前一步的输出
- **AI 介入点**：脚本无法完成的部分由 AI 处理
- **结果合并**：并行结果全部返回后统一组装
